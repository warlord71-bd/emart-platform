# Emart Queue and State Contract

Version: 2026-06-25-v1
Scope: durable queues, state files, ledgers, and generated output used by Emart automation. Documentation only; this file does not change runtime behavior.

## Shared Contract

Every queue/state owner should follow these rules when implementing or repairing workers:

1. Write state atomically: write to a temporary file in the same directory, fsync/close where practical, then rename.
2. Treat JSON parse failure as fatal for that state file. Preserve the corrupt file as `.corrupt-YYYYMMDD-HHMMSS` before rebuilding from source inputs.
3. Use idempotency keys for anything that can publish, write Woo/WordPress, send Telegram, or record performance. Preferred key: stable job/campaign/action ID plus platform.
4. Keep append-only audit ledgers for publish/apply outcomes. Generated scratch output may be ignored by git; human approval queues and schemas stay versioned.
5. Use one active writer per domain. If a second writer is needed, add a lock file with PID, hostname, started_at, expires_at, and command.
6. Dead-letter instead of deleting. Failed or rejected jobs move to a failed/rejected directory with reason, timestamp, and recovery note.
7. Retention defaults: keep source manifests, approval records, result ledgers, and rollback files indefinitely unless owner approves archival; rotate bulky generated media/output to `/root/.attic-YYYY-MM-DD/` after the campaign/review window.

## Required Future Schema Additions

- Add `schema_version` to every new state file and every queue job.
- Add `created_at`, `updated_at`, `writer`, `source_commit`, and `idempotency_key` to publish/apply-capable jobs.
- Add `retry_count`, `max_retries`, `last_error`, `next_retry_at`, and `dead_letter_reason` to queue jobs.
- Add `approval` object for all jobs that can publish or write externally: `{state, approver, channel, message_id, approved_at}`.
- Add `rollback` object for all jobs that write WordPress/Woo/storefront content.
